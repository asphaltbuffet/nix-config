# NixOS Configuration Development

## Nix Language Essentials

For detailed Nix language patterns, see [references/nix_language_patterns.md](references/nix_language_patterns.md).

Key points for writing Nix in this codebase:

- **Formatter**: alejandra (not nixpkgs-fmt or nixfmt). Run `just fmt` — it also
  applies statix and deadnix autofixes
- **Linter**: statix + deadnix (check-only). Run `just lint`
- **Style**: 2-space indentation, no trailing commas in attrsets, `{` on same line as function args
- **Strings**: Use double quotes for simple strings, `''` for multi-line. Interpolation: `${expr}`
- **Paths**: Bare paths like `./foo/bar.nix` are valid expressions (no quotes needed)

## Build and Validation Commands

All commands use `just` (justfile at repo root). Host defaults to current hostname (via `$(hostname)`).

| Command | Purpose |
|---------|---------|
| `just build` | Build without activating |
| `just switch` | Build and activate (sets boot default) |
| `just test` | Build and activate without boot default |
| `just lint` | Check formatting + statix + deadnix (read-only) |
| `just fmt` | Format + apply statix/deadnix autofixes |
| `just warnings` | Report Nix evaluation warnings for a host |
| `just precommit` | `just lint` + `nix flake check` |
| `just update` | Update flake.lock inputs |
| `just inputs` | Show flake inputs and versions |
| `just hosts` | List available host names |

`just build`/`switch` use `nh`, which prints a package diff against the current
system automatically — there is no separate diff recipe.

To build for a specific host: `just build kushtaka`

After any change, validate with `just build` and `just lint` before committing. Formatting errors cause CI failures.

## Repository Architecture

```
flake.nix                       # Entry point: inputs, mkHost, devShell, formatter
nixos/
  hosts/<hostname>/             # Per-machine config (auto-discovered by flake)
    configuration.nix           # Imports profiles + common modules, sets hostname
    hardware-configuration.nix  # Generated by nixos-generate-config
  profiles/                     # Composable system-level feature sets
    base.nix                    # All systems: boot, locale, nh, home-manager setup
    server.nix                  # Headless home-lab overlay (no GUI/desktop)
    gaming.nix                  # Gaming services (Steam, gamemode)
    laptop/                     # Hardware-specific (KDE, power, t14/x1carbon), GUI
  installer/                    # Bootable ISO config + nixos-bootstrap helper
  common/                       # Shared NixOS modules (single-concern)
    users.nix                   # User definitions + home-manager.users mappings
    tailscale.nix               # VPN with agenix secret
    1password.nix, firefox.nix  # App-specific system config

home/
  users/<username>.nix          # Per-user config: imports roles, sets identity,
                                  and env vars
  roles/                        # Composable user-level feature sets
    base.nix                    # Core CLI tools, shell, GUI apps
    dev.nix                     # Development tools (nvim, jj, mise, claude, ...)
    admin.nix                   # Network/sysadmin tools
    player.nix                  # Gaming applications
  modules/<tool>/default.nix    # Per-tool home-manager configuration

secrets/
  secrets.nix                   # Declares secrets and authorized keys
  *.age                         # Encrypted secret files (agenix)
```

### Key Architecture Patterns

**Host auto-discovery**: The flake reads `nixos/hosts/*` directory names and generates `nixosConfigurations` automatically. Adding a directory with `configuration.nix` is sufficient.

**Layered composition**: Hosts import profiles, profiles import common modules. Users import roles, roles import tool modules. This creates a clean dependency tree:

```
host/configuration.nix → profiles → common modules
user.nix → roles → tool modules
```

**User-host binding**: `nixos/common/users.nix` maps system users to home-manager configs via `home-manager.users.<name> = import ../../home/users/<name>.nix`.

**Special args**: The flake passes all inputs as `specialArgs`, so any module can access `inputs.agenix`, `inputs.nixos-hardware`, NUR packages, etc.

## Adding New Components

### New Host

1. Create `nixos/hosts/<hostname>/configuration.nix` importing needed profiles
2. Generate `hardware-configuration.nix` with `nixos-generate-config` (during
   install bootstrap on new host)
3. Auto-discovered by flake — no flake.nix edits needed

### New Home-Manager Module

1. Create `home/modules/<tool>/default.nix`
2. Use `programs.<tool>.enable = true;` or `home.packages` pattern
3. Import from the appropriate role file in `home/roles/`

Module template:
```nix
{pkgs, ...}: {
  programs.toolname = {
    enable = true;
    # configuration here
  };
}
```

### New User

1. Create `home/users/<name>.nix` importing desired roles
2. Ask user for `uid` and `gid` that matches NAS setup.
3. Add user to `nixos/common/users.nix` (both `users.users` and `home-manager.users`)

1. Add entry to `secrets/secrets.nix` with `publicKeys = users ++ systems;`
2. Encrypt the file: `nix shell "github:ryantm/agenix" --command agenix -e secrets/<name>.age`
3. For a **user** secret exposed as an env var, add a `{secret, env}` record to the
   `secretEnvs` list in `home/users/<name>.nix` — `age.secrets` and the zsh export
   are derived from it. For a **system/module** secret, reference it directly:
   `age.secrets.<name>.file = ../../secrets/<name>.age;`
4. After adding a new host or user key to `secrets.nix`: `just rekey`

**Note**: System secrets may require `sudo agenix -e <name>.age -i /etc/ssh/ssh_host_ed25519_key`

See `docs/references/secrets-sop.md` for the full workflow.

## Common Nix Patterns in This Repo

### Module argument pattern
```nix
{
  pkgs,
  lib,
  config,
  inputs,
  ...
}: {
  # module body
}
```
Always include `...` to accept extra args. Only destructure what the module uses.

### Conditional defaults with mkDefault
```nix
services.openssh.enable = lib.mkDefault true;  # hosts can override to false
```

### NUR package access
```nix
# In modules that need NUR packages:
home.packages = with pkgs; [
  nur.repos.goreleaser.goreleaser-pro
];
```

### Agenix secret consumption
```nix
# Declare in user config:
age.secrets.mykey.file = ../../secrets/mykey.age;

# Use the decrypted path:
programs.zsh.initContent = ''
  MY_KEY="$(< "${config.age.secrets.mykey.path}")"
  export MY_KEY
'';
```

## Code Navigation (Serena)

Prefer Serena's semantic tools over reading whole files — they are more
token-efficient and understand symbol relationships. In this environment Serena
is loaded via a plugin, so tools are namespaced
`mcp__plugin_claude-code-home-manager_serena__<name>`.

| Task | Tool |
|------|------|
| Find a function/option definition | `find_symbol` |
| List all symbols in a file | `get_symbols_overview` |
| Find all importers/callers of a symbol | `find_referencing_symbols` |
| Rename a binding across files | `rename_symbol` |
| Replace a module/function body | `replace_symbol_body` |

This plugin build does **not** expose `list_dir`, `find_file`, or
`search_for_pattern` — for directory browsing, file-finding, and text search use
the built-in Glob/Grep tools (or `fd`/`rg` via Bash).

**Never read an entire `.nix` file to locate a binding** — use
`get_symbols_overview` or `find_symbol` first, then read only the relevant body.

## Formatting Rules

alejandra enforces specific formatting. Notable conventions:
- Function args: `{...}: {` (opening brace on same line)
- List items and attrset entries: no trailing commas
- Multiline lists: one item per line, closing `]` or `}` on own line
- `let ... in` blocks: bindings indented 2 spaces
- String interpolation in paths: use `${}` syntax

When unsure about formatting, write the code and run `just fmt` to auto-fix.
